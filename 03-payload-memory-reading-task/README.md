# Task 3 — Payload Memory Reading + NGHam Packet Encoding Pipeline

## What this task was asking for

This is where Task 1 (Python NGHam study) and Task 2 (FreeRTOS notes) actually come together.
The goal: build a 3-task FreeRTOS pipeline in C that simulates the real onboard flow for
handling occultation event data (mock data standing in for actual payload memory reads) —
fragment it into chunks, pass it through a queue manager, and encode each chunk into a real
NGHam frame, exactly like it would need to happen on the actual MSP430 before transmission.

This runs on the **FreeRTOS POSIX simulator** for now (via WSL on my machine) rather than on
real MSP430 hardware — the idea is to get the task/queue/encoding logic fully correct and
tested first, before dealing with actual hardware registers and the CCS toolchain.

### Build system

The `Makefile` compiles each `.c` file into its own `.o` object file first (`main.c`, the
FreeRTOS kernel sources like `tasks.c`/`queue.c`/`list.c`/`timers.c`/`event_groups.c`, the
`heap_3.c` memory manager, and the POSIX port files `port.c`/`wait_for_event.c`), then links
all the object files together into `build/cubesat_pipeline`. `-D_POSIX_C_SOURCE=200809L` is
needed because the FreeRTOS POSIX port relies on some POSIX-specific functions (like certain
`pthread` and timing calls) that aren't exposed by default under strict C standards, and
`-lpthread` is required at link time since the POSIX port simulates each FreeRTOS task as an
actual pthread under the hood. To build and run:
```
make run
```
which builds everything and then immediately executes `build/cubesat_pipeline`.

## `payload_chunk.h` — the shared data structure

```c
typedef struct {
    uint16_t occultation_id;
    uint8_t  sequence_number;
    uint32_t timestamp;
    uint8_t  total_chunks;
    uint8_t  data_size;
    uint8_t  data[CHUNK_DATA_SIZE];
} PayloadChunk_t;
```

This struct is the "unit of currency" that flows through both queues in the pipeline. Every
field exists for a reason:
- `occultation_id` — which event this chunk of data belongs to (since one event might get
  split across several chunks, I need a way to know which chunks belong together)
- `sequence_number` and `total_chunks` — so the receiving side (ground station, eventually)
  knows this is chunk 3 out of 5, for example, and can reassemble things in order
- `timestamp` — when this chunk was generated, using `xTaskGetTickCount()` from FreeRTOS
- `data_size` — how many bytes in `data[]` are actually valid, since the last chunk of an
  event is usually smaller than `CHUNK_DATA_SIZE`
- `data[CHUNK_DATA_SIZE]` — the actual payload bytes, fixed at 32 bytes per chunk. The
  comment says "matches Amrit's approach" because this chunk size was agreed on with a
  teammate so our modules interoperate correctly.

Why fixed-size chunks instead of variable-length? Because FreeRTOS queues need to know the
exact size of each item up front (`xQueueCreate(10, sizeof(PayloadChunk_t))`) — queues copy
fixed-size blocks, they can't hold "however many bytes happen to be in this particular chunk."
So fragmenting into a known max size was the natural choice.

## `main.c` — the three tasks

### Task 1: `vTask1_PayloadMemReader` — reading and fragmenting

This task simulates reading payload memory for a set of mock occultation events:
```c
static const MockEvent_t mock_events[] = {
    { .occultation_id = 1, .data_size = 100 },
    { .occultation_id = 2, .data_size = 45  },
    { .occultation_id = 3, .data_size = 70  },
};
```
These stand in for real payload memory reads (which would eventually come from actual flash
or SRAM on the satellite) — for now I fill a buffer with predictable fake bytes
(`0x10 + (e * 0x30) + i`) just so I can visually confirm chunk-splitting is working correctly
in the printed output.

For each event, I calculate how many chunks it needs:
```c
uint8_t total_chunks = (data_len + CHUNK_DATA_SIZE - 1) / CHUNK_DATA_SIZE;
```
This is the standard "ceiling division" trick — adding `CHUNK_DATA_SIZE - 1` before dividing
makes sure that if there's a remainder (like 100 bytes / 32 = 3 remainder 4), it still counts
as needing 4 chunks rather than rounding down to 3 and losing data.

Then I loop over `seq` from 1 to `total_chunks`, building one `PayloadChunk_t` per chunk,
filling in all its metadata, copying the right slice of `mock_payload` into `chunk.data`, and
pushing it into `xQueue1_RawChunks`. I added `vTaskDelay(pdMS_TO_TICKS(50))` between each send
purely so the printed console output is readable in real time instead of flooding the terminal
instantly — this wouldn't be needed in the real firmware.

Why not fragment and send everything for an event in one queue call? Because Task 2 (and
eventually the radio) needs to process one chunk at a time anyway — NGHam frames themselves
are always chunked messages, so this design naturally lines up with how the actual downlink
would eventually work: one NGHam frame per queue item.

### Task 2: `vTask2_QueueManager` — the FIFO manager

```c
if (xQueueReceive(xQueue1_RawChunks, &chunk, pdMS_TO_TICKS(3000)) == pdTRUE)
{
    total_forwarded++;
    ...
    xQueueSend(xQueue2_OrderedChunks, &chunk, portMAX_DELAY);
}
```
Right now this task is intentionally simple — it just receives from Queue 1 and forwards into
Queue 2, logging what it forwarded (occultation id, sequence number, size, timestamp) along
the way. I kept it as its own separate task (rather than having Task 1 write directly to Queue
2) because this is exactly where reordering logic or priority-based scheduling between
occultation events could go later, without having to touch Task 1 or Task 3 at all. Keeping
this as a clean middle stage means the pipeline is easier to extend.

The `else` branch — if nothing arrives within 3 seconds — assumes Task 1 has finished
producing data, prints a final count, and puts the task into an infinite idle loop
(`for (;;) vTaskDelay(...)`). This is a simple way to detect "the pipeline is done" without
needing a separate signaling mechanism, since I know Task 1 stops sending once it's out of
mock events.

### Task 3: `vTask3_NGHamEncoder` — building the actual NGHam frame

This is the task where the payload chunk actually becomes a real over-the-air-shaped packet.

**Step 1 — serialize the chunk into a flat byte buffer:**
```c
ngham_payload[p++] = (uint8_t)(chunk.occultation_id >> 8);
ngham_payload[p++] = (uint8_t)(chunk.occultation_id & 0xFF);
ngham_payload[p++] = chunk.sequence_number;
...
memcpy(&ngham_payload[p], chunk.data, chunk.data_size);
```
NGHam doesn't know anything about my `PayloadChunk_t` struct — it just wants a flat array of
bytes. So before encoding, I manually pack each metadata field in big-endian order (most
significant byte first, which is the networking convention) followed by the actual chunk data.
I do the 16-bit `occultation_id` and 32-bit `timestamp` splits manually with bit shifts
(`>> 8`, `>> 16`, etc.) rather than just doing a raw `memcpy` of the struct, because struct
padding and endianness aren't guaranteed to match across compilers/architectures — manually
serializing avoids that whole class of bug.

**Step 2 — actually build the NGHam packet:**
```c
int packet_len = ngham_build_packet(ngham_payload, (uint8_t)p, ngham_packet);
```
This calls into `ngham.h` (explained below) which does the real protocol work: picking a size
class, computing the header, CRC, padding, and Reed-Solomon parity, and stitching the whole
frame together with preamble and sync word.

**Step 3 — print the frame broken into its actual protocol fields**, so I (and anyone
reviewing this) can visually verify every part of the frame is where it should be: preamble,
sync word, size tag, header byte, payload, CRC, padding, and RS parity — printed as separate
labeled hex blocks rather than one giant hex dump. This was genuinely useful for debugging —
I could immediately tell if my byte offsets were wrong because a field would clearly contain
the wrong kind of data (like payload bytes showing up in what should've been the CRC section).

Same "no more chunks" idle pattern as Task 2 in the `else` branch, tracking `total_encoded`.

### `main()` — wiring it all together

```c
xQueue1_RawChunks     = xQueueCreate(10, sizeof(PayloadChunk_t));
xQueue2_OrderedChunks = xQueueCreate(10, sizeof(PayloadChunk_t));

xTaskCreate(vTask1_PayloadMemReader, "Task1", configMINIMAL_STACK_SIZE * 2, NULL, 1, NULL);
xTaskCreate(vTask2_QueueManager,     "Task2", configMINIMAL_STACK_SIZE * 2, NULL, 1, NULL);
xTaskCreate(vTask3_NGHamEncoder,     "Task3", configMINIMAL_STACK_SIZE * 4, NULL, 1, NULL);

vTaskStartScheduler();
```
Both queues hold up to 10 `PayloadChunk_t` items. All three tasks get equal priority (1) since
none of them needs to preempt the others for this test — they just each wait on their queue
and do their job when data shows up. Task 3 gets double the stack of the other two
(`configMINIMAL_STACK_SIZE * 4`) because it needs local buffers for `ngham_payload[64]` and
`ngham_packet[300]`, which are bigger than what Tasks 1 and 2 need on their own stacks.

`vTaskStartScheduler()` hands control over to FreeRTOS permanently — code after it (`for(;;)`)
never actually runs under normal conditions, it's just there because `main()` technically
needs a return path if the scheduler ever failed to start (e.g. ran out of heap for the idle
task).

### The three hook functions at the bottom

- `vAssertCalled` — backs the `configASSERT` macro from `FreeRTOSConfig.h`. If any internal
  FreeRTOS sanity check fails, this prints exactly which file and line triggered it instead of
  just silently crashing.
- `vApplicationMallocFailedHook` — fires if FreeRTOS's heap runs out of memory when trying to
  allocate a task's stack or a queue's storage.
- `vApplicationStackOverflowHook` — fires if a task overflows its allocated stack, printing
  which task name specifically blew its stack (relevant to how I sized Task 3's stack above).

These aren't optional boilerplate — `FreeRTOSConfig.h` explicitly turns on
`configCHECK_FOR_STACK_OVERFLOW` and `configUSE_MALLOC_FAILED_HOOK`, so the kernel expects
these functions to exist somewhere, or the build won't link.

## `ngham.h` — the actual NGHam protocol implementation

This is a from-scratch C implementation of the pieces of NGHam needed to build a valid frame,
written after studying both the TTC 2.0 docs and how `pyngham` behaves (Task 1).

### CRC16-CCITT
```c
static uint16_t ngham_crc16(const uint8_t *data, int len)
```
Standard bit-by-bit CRC16-CCITT: starts at `0xFFFF`, XORs in each byte at the top of the
register, shifts left 8 times per byte (XOR-ing with the polynomial `0x1021` whenever the top
bit is set), and XORs the final result with `0xFFFF` at the end. This matches the CRC scheme
specified for NGHam frames, and it's what protects the header+payload portion of the packet
against corruption (separately from Reed-Solomon, which protects the whole coded block).

### The size table
```c
static const NGHamSizeConfig_t ngham_size_table[7] = { ... };
```
NGHam doesn't use one fixed frame size — it picks from **7 predefined size classes**, each
with its own 3-byte size tag, RS(n,k) parameters, and max payload capacity. Smaller payloads
get a smaller frame (less overhead, but also less RS protection since `n-k` is smaller);
larger payloads get bigger frames with proportionally more parity bytes. `ngham_build_packet`
picks the smallest size class that can actually fit the payload, which keeps overhead as low
as possible for whatever's being sent.

### Reed-Solomon over GF(256)

This is genuinely the trickiest part of the whole file, so I want to be upfront about
something important: **the field polynomial used here (`0x11D`) is a common textbook/demo
GF(256) polynomial, not necessarily the exact one specified in the official NGHam
spec (which the code comment notes uses `0x187` with a specific generator root)**. That means
this encoder is *structurally* a real, working Reed-Solomon systematic encoder — it correctly
implements the RS algorithm — but it won't necessarily produce byte-identical output to the
official reference NGHam C/Python implementation unless the field polynomial and generator
root are matched exactly. That's flagged directly in the code comment and is the next thing
to fix before this needs to interoperate with a real ground station decoder.

With that caveat in mind, here's how it works:
- `gf_init()` builds log/antilog tables for GF(256) arithmetic — this lets multiplication in
  the field be done as `gf_exp[gf_log[a] + gf_log[b]]` instead of needing actual polynomial
  multiplication + modular reduction every time, which is much faster and simpler on an
  embedded target.
- `rs_generator_poly()` builds the RS generator polynomial for however many parity symbols
  (`nsym`) are needed, using the standard iterative construction (multiply the running
  polynomial by `(x - gf_exp[i])` for each root).
- `rs_encode()` does systematic RS encoding — it runs the data bytes through a
  polynomial-division-style shift register (the "feedback" pattern), and what's left in the
  register at the end is the parity. "Systematic" means the original data bytes appear
  unchanged in the output, with parity bytes appended after — which is exactly what
  `ngham_build_packet` needs, since the payload has to survive as human/software-readable
  data even before RS decoding happens on the receiving end.

### `ngham_build_packet()` — assembling the full frame

Step by step:
1. **Pick a size class** — smallest one where `payload_len <= max_payload`.
2. **Compute padding** — `k - 3 - payload_len`, since `k` (the RS data block size) has to be
   filled exactly: 1 header byte + payload + 2 CRC bytes + padding = `k`.
3. **Build the header byte** — currently just holds the padding size (`padding_size & 0x1F`,
   masked to 5 bits since NGHam headers only reserve that many bits for it).
4. **Compute CRC** over header+payload only (not the padding — the padding is filler, not
   meaningful data, so it isn't included in the integrity check).
5. **Append zero padding** until the data block is exactly `k` bytes.
6. **Run Reed-Solomon** on the full `k`-byte data block to get `n - k` parity bytes.
7. **Stitch the final frame together**: 4-byte preamble (`0xAA` repeated — a bit pattern that
   helps the radio's clock/data recovery lock onto the signal before real data starts), 4-byte
   sync word (`0x5D 0xE6 0x2A 0x7E` — a fixed pattern marking "real frame starts here"),
   3-byte size tag (tells the receiver which size class this is, so it knows how many bytes to
   expect and which RS parameters to use), then the full data block, then the RS parity.

This function returns `-1` if the payload is too big for even the largest size class — Task 3
checks for this and logs an error rather than silently building a broken frame.
